# 需求发现防跑偏检查

## 目标

确保 `bkn-requirement` 始终服务于需求发现，而不是在 PRD 阶段提前滑向技术设计、建模设计或机器消费结构。

## 三条红线

1. **PRD 主体必须使用业务语言。**
   - 业务用户应当无需理解 BKN、对象类、关系类、Action、字段映射或机器 schema，就能评审文档。
2. **“业务已确认内容”只能写已被业务材料支撑的事实。**
   - 带有“待确认”“候选”“范围未定”的内容，不能混入已确认结论。
3. **需求发现阶段不输出机器可读 schema。**
   - PRD 可以为后续建模做准备，但不能因此把自己写成建模输入文件。

## 最终输出前自检

### A. 是否重新技术化

- 是否把“用户要解决什么问题”写成了“系统要建什么对象 / 接口 / Action”？
- 是否在正文主叙事中出现了 `object_type`、`relation_type`、`ActionType`、字段映射、关系基数、data_view、operator 等词？
- 如果出现了，它们是否已被转译为业务动作、业务判断、责任边界或验收样例？
- 一个不懂 BKN 的业务专家，是否仍能独立审阅这份 PRD？

### B. 是否把候选伪装成确认

- “业务已确认内容”中的每一条，是否都能回指到至少一个场景、业务规则、验收用例或原始材料？
- 是否存在这样的表述：句子前半段像结论，后半段却带“待确认 / 范围待定 / 字段未确认”？
- 若某项只是在工程上很可能成立，但业务尚未确认，是否已移入“仍属建模候选”或“需下游建模阶段判定的问题”？



### D. 是否通过确认状态门禁

- `业务已确认内容` 中是否出现“待确认 / 未确认 / 待对齐 / 是否 / 范围待定 / 范围待确认 / 候选”？
- 如果出现，必须拆分：已被业务材料支撑的部分留在“业务已确认内容”；未确认口径、字段、容差、范围、执行方式移入“仍属建模候选”或“需下游建模阶段判定的问题”。
- 带括号的补充说明同样适用，例如“字段待对齐”“容差未确认”“是否自动下单未确认”都不得留在“业务已确认内容”。

### E. 是否通过本体一致性门禁

- 关系层出现的每个核心名词，是否已在概念模型层逐项说明“是什么 / 为什么需要”？
- 动力层输出的结果，是否已判断为业务对象、派生结果或状态？
- `满足方案行 / 缺口行 / 报表行 / 看板指标` 是否被默认当作对象？若没有稳定身份、生命周期、责任人或后续业务动作，应写为结果候选或输出明细，而不是对象候选。
- 关系层不能引入概念层没有交代的核心对象；如果确实需要，应先补概念层。

### C. 是否发生 schema 回潮

- PRD 中是否出现 YAML / JSON 机器交接块？
- `BKN_Creator` 交接摘要是否仍是中文业务摘要，而非结构化机器输入？
- 是否因为担心“后续不好消费”，就在需求发现阶段提前塞入机器字段？

## 反例与正例

### 1. 技术化反例

**错误：**

> 建立 `Demand`、`Material`、`PurchaseOrder` 三类对象，设计 `calculate_fulfillment` Action，输出 shortage_view。

**正确：**

> 当新的产品需求进入时，PMC 需要判断现有库存、在途采购和可拆库存是否足以支撑交付，并看清缺口来自哪里。

### 2. 已确认混入候选反例

**错误：**

> 在途 PO 晚于需求交期时不计入可用（容差天数待确认）。

**正确：**

- 业务已确认内容：在途 PO 是否可支撑需求，需要结合需求交期判断。
- 仍属建模候选：PO 延迟容差天数规则。

### 3. schema 回潮反例

**错误：**

> 在 PRD 末尾追加机器字段、英文结构化块或下游程序消费用 schema。

**正确：**

> 在 PRD 末尾保留中文交接摘要：按场景说明业务对象、业务联系、业务逻辑、责任控制，再归并为“业务已确认内容 / 仍属建模候选 / 需下游建模阶段判定的问题”。

## 失败信号

出现以下任一情况，都说明文档已经开始跑偏，应回退重写：

- 业务读者要先理解建模概念才能读懂 PRD；
- 一条“已确认内容”里同时出现了“待确认”；
- PRD 末尾开始长出 YAML / JSON 机器块；
- 文档越来越像 BKN 设计稿，而不是业务需求文档；
- 团队开始为了“后续好建模”，牺牲当前业务表达的清晰度。


### F. 是否发生本体建模漂移

- 是否把编码、字段口径、统计维度、余额、数量、指标、结果、明细、行、报表直接升格为对象？
- 是否把 `SKU`、`库存余额`、`可用库存`、`满足方案行`、`缺口行`、`查询结果`、`分析报表` 当作对象？
- 若命中，应回到 `references/bkn-ontology-modeling-method.md` 重新分类：对象、属性、枚举、关系、动力逻辑、结果、治理要求。
- 结果类、粒度类和扣减类内容只有能说明稳定身份、生命周期、责任人、独立管理动作或后续业务动作，才可升格为对象候选。


### G. 是否过度拆分核心对象

- 是否把同一核心对象的粒度、状态、数量、口径直接拆成多个对象？
- 例如库存场景中，`库存` 应优先作为核心对象；批次、状态、数量、占用数量、占用来源默认作为属性、枚举、计算口径或关系处理。
- 只有业务按批次或占用事实独立追踪、冻结、释放、审批、盘点、审计时，才把 `库存批次` 或 `库存占用` 升格为对象候选。


### H. 是否把规则或场景角色误建成对象

- 是否把 `替代规则`、`分配规则`、`可用量口径` 这类规则 / 口径直接放进对象层？
- 是否把 `拆料来源产品`、`候选物料`、`优先需求` 这类场景角色名当成新对象？
- 正确处理：规则优先进入关系约束、逻辑属性、Function / operator 候选或治理要求；场景角色名应回到已有对象，例如产品、物料、库存、客户需求。
- 只有规则本身具备编号、版本、生效期、审批、发布废止、责任人和审计生命周期，才可作为规则类对象候选。
