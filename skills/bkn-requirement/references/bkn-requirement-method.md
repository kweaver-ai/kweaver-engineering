# BKN 需求发现方法

用于将业务材料转换为业务场景中心的 BKN-ready PRD。

## 成熟度

| 等级 | 含义 | 建议路线 |
|---|---|---|
| R0 Idea | 只有粗略想法 | `hold` |
| R1 Use Case Brief | 用户、目标、场景大致明确 | `interview` |
| R2 PRD Candidate | 有场景、流程和规则候选，但关键口径待确认 | `prd_refinement` |
| R3 PRD Ready | 业务场景、规则、系统数据和验收用例基本可评审 | `business_review` |
| R4 Implementation Ready | 字段、样例、权限、交互、验收和 handoff 已确认 | `bkn_creator_handoff` |

## 主线方法

不要从 BKN 元素开始问。先还原业务工作，再把候选建模项放到末尾交接。

```text
业务目标
→ 业务用户
→ 业务场景
→ 当前流程 / 目标流程
→ 输入 / 输出
→ 业务规则 / 判断标准
→ 系统 / 表单 / 数据
→ 权限 / 审批 / 异常
→ 界面 / 交互期望
→ 业务验收用例
→ BKN_Creator 交接摘要
```

## 场景拆解

每个场景至少记录：

- 场景名称；
- 主要用户；
- 触发方式；
- 当前流程；
- 目标流程；
- 业务输入；
- 业务输出；
- 业务规则；
- 异常边界；
- 人工确认点；
- 界面 / 交互期望；
- 业务验收标准。

如果材料里只有“对象、关系、指标、Action”，需要反向追问：这些元素服务哪个业务场景、谁使用、在哪个流程节点使用、怎样验收。

## 业务规则

业务规则优先使用业务语言表达，例如：

- 可用库存如何扣减；
- 风险等级如何判断；
- 供应关系方向如何解释；
- 哪些动作只能生成草案；
- 哪些情况必须拒绝或转人工审批；
- 哪些数据缺失时不能给确定性结论。

不要在 PRD 主体中把业务规则写成 `operator` 或执行工厂算子。若后续确实需要，应放入 `BKN_Creator` 交接摘要的 `candidate_logic_properties`。

## 系统与数据发现

必须记录：

- 涉及的业务系统、表单、报表、Excel、API、数据库或数据视图；
- 是否为事实来源；
- 数据负责人；
- 关键字段或业务信息；
- 对象 ID 和跨系统映射问题；
- 数据质量风险；
- 延迟和刷新频率；
- 访问限制；
- 外部写回系统。

不要让表结构决定 BKN 对象模型。数据材料用于验证或实现业务对象。

## BKN_Creator 交接

PRD 末尾输出 `bkn_creator_handoff`，并分三层：

| 层级 | 内容 |
|---|---|
| `business_confirmed` | 业务专家可直接确认的场景、业务对象、规则、系统和验收用例。 |
| `candidate_only` | AI 工程师从 PRD 推导出的候选对象、关系、逻辑属性、Action、Skill、数据视图。 |
| `needs_bkn_creator_decision` | 需要下游决定的建模问题，如对象边界、关系对象、逻辑属性挂载、Action 执行模式。 |

## 候选建模线索

这些线索只作为后台检查和 handoff 候选，不应喧宾夺主：

- 业务名词：候选对象；
- “属于、包含、供应、影响、依赖、拥有”：候选关系；
- “评分、优先级、风险、影响范围、排序、推荐、模拟”：候选对象逻辑属性；
- “创建、提交、审批、关闭、发布、提醒、写回”：候选 Action；
- “必须审批、不能自动、只能草案、需要审计”：治理要求；
- “问答、分析、建议、解释、总结”：候选 Skill / Agent 能力。
