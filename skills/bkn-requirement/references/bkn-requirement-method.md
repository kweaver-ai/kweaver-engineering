# BKN 需求发现方法

用于将客户背景、调研目标、AI 会议纪要、业务材料和上一版 PRD 持续转换为业务场景中心的 BKN-ready PRD。

## 需求发现生命周期

需求发现不是一次性生成 PRD，而是多轮收敛：

```text
Intake
→ Research Plan
→ Interview
→ Meeting Digest
→ PRD Iteration
→ Readiness Gate
→ Handoff
```

| 阶段 | 输入 | 输出 |
|---|---|---|
| Intake | 客户名称、行业、部门、粗略方向 | 最小澄清问题、调研前置条件 |
| Research Plan | 客户背景、拜访目标、已有材料 | 客户现场短提纲、参会角色建议、材料索要清单 |
| Interview | 现场访谈、线上会议 | AI 会议纪要、录音转写、白板和材料附件 |
| Meeting Digest | 豆包 / 飞书 / 腾讯会议等会议纪要 | 本轮新增事实、假设、冲突、场景、规则、待确认问题 |
| PRD Iteration | 调研大纲、会议纪要及材料、上一版 PRD | 新一版 PRD、修订摘要、版本记录、下一轮追问 |
| Readiness Gate | PRD、样例、规则、系统数据 | R0-R4 成熟度和缺口判断 |
| Handoff | 已确认 PRD | BKN_Creator 交接摘要 |

客户现场模板应短、业务化、开放式；技术字段、建模线索和系统细表应进入会后结构化或 handoff，不应压到客户访谈现场。

## 项目资料组织

每个客户或项目应有独立目录，默认：

```text
docs/requirements/prj-<客户或项目简称>/
```

该目录中存放本项目所有调研大纲、会议纪要、验证输出、PRD 版本、补充材料和 handoff。不要把多个项目的材料混放在一个根目录。

命名规则：

| 文档类型 | 命名格式 | 示例 |
|---|---|---|
| 调研大纲 | `<项目名>-第X轮调研大纲.md` | `国泰海通券结业务-第1轮调研大纲.md` |
| 调研备忘 | `YYYYMMDD-<项目名>-第X轮现场交流调研备忘.md` | `20250515-国泰海通-第1轮现场交流调研备忘.md` |
| 验证输出 | `<项目名>-prd-第X轮验证输出.md` | `国泰海通券结业务-prd-第1轮验证输出.md` |
| PRD | `<项目名>-PRD vX.Y.md` | `国泰海通券结业务-PRD v0.1.md` |

每轮输入材料应放入独立轮次目录：

```text
docs/requirements/prj-<客户或项目简称>/inputs/round-XX/
```

推荐命名：

| 输入类型 | 命名格式 | 示例 |
|---|---|---|
| 调研大纲 | `YYYYMMDD-<项目名>-第X轮-input-调研大纲.md` | `20250515-国泰海通券结业务-第1轮-input-调研大纲.md` |
| 会议纪要 | `YYYYMMDD-<项目名>-第X轮-input-会议纪要.md` | `20250515-国泰海通券结业务-第1轮-input-会议纪要.md` |
| 客户材料 | `YYYYMMDD-<项目名>-第X轮-input-客户材料-<材料名>.<ext>` | `20250515-国泰海通券结业务-第1轮-input-客户材料-交易所备案流程.pdf` |

原则：

- 不移动用户原始文件；
- 输入文件不在项目目录内时，必须在生成验证输出或 PRD 前复制到本轮 `inputs/round-XX/`；
- 输入文件已在项目目录内时，可以不复制，但要登记；
- 每轮生成或更新 `inputs/round-XX/source-manifest.md`；
- 验证输出和 PRD 都必须列出“本轮输入来源”。

`source-manifest.md` 至少记录：原始路径、归档路径、材料类型、所属轮次、用途、处理时间、是否实际用于本轮输出。

归档完整性规则：

- manifest 只能记录真实发生的归档动作；
- `是否复制=是` 或 `copied_to_project=true` 时，归档路径必须真实存在；
- 如果复制失败、权限不足或路径无法访问，必须记录为 `是否复制=否`，并写明原因；
- 不得为了让清单看起来完整而填写不存在的归档路径；
- 最终验证输出或 PRD 开头必须列出本轮输入来源；若归档未完成，必须显式提示。

## 项目上下文自动识别

用户只提供项目目录时，不要求用户补齐所有文件。Skill 应先自动识别最新项目上下文，再根据用户意图选择模式。

自动识别顺序：

1. 最新 PRD：最高版本号的 `<项目名>-PRD vX.Y.md`；
2. 最新验证输出：最高轮次的 `<项目名>-prd-第X轮验证输出.md`；
3. 最新调研大纲、会议纪要和 `inputs/round-XX/` 材料；
4. 用户意图为“处理会议纪要 / 看本轮调研”时，进入 `meeting_digest_mode`；
5. 用户意图为“更新 PRD / 迭代需求”时，进入 `prd_iteration_mode`；
6. 用户意图为“整理需求 / 标准化 PRD / 改写 PRD / 输出正式文档”时，进入 `prd_mode`；
7. 用户意图为“评估 PRD / 评审 PRD / 检查 PRD / 是否可进入 BKN_Creator”时，进入 `review_mode`；
8. 用户意图为“准备拜访”时，基于上一轮验证输出或上一版 PRD 生成下一轮调研大纲。

只有存在多个同版本 PRD、多个同轮次会议纪要、轮次无法判断或用户意图冲突时，才要求用户确认。

第一轮后不一定直接生成 PRD：

- 如果信息不足以形成 PRD，输出验证性 `meeting_digest`，用于判断缺口和下一轮调研重点。
- 如果信息足以形成候选 PRD，输出 `PRD v0.1`，状态标注为 `Draft`，并保留未确认事项。

## 成熟度

| 等级 | 含义 | 建议路线 |
|---|---|---|
| R0 Idea | 只有粗略想法 | `hold` |
| R1 Use Case Brief | 客户、行业、用户、目标或场景大致明确 | `research_plan` / `interview` |
| R2 PRD Candidate | 有场景、流程和规则候选，但关键口径待确认 | `prd_iteration` / `prd_refinement` |
| R3 PRD Ready | 业务场景、规则、系统数据和验收用例基本可评审 | `business_review` |
| R4 Implementation Ready | 字段、样例、权限、交互、验收和 handoff 已确认 | `bkn_creator_handoff` |

## 输出模式选择

| 输入状态 | 模式 | 原则 |
|---|---|---|
| 弱输入，仅有客户、行业、部门、拜访目的 | `intake_mode` | 只问最少澄清问题，不输出 PRD。 |
| 拜访前准备 | `research_plan_mode` | 输出 1-2 页客户现场短提纲。 |
| 输入 AI 会议纪要 | `meeting_digest_mode` | 自动提取本轮调研更新，不要求手工填表。 |
| 调研大纲 + 会议纪要及材料 + 上一版 PRD | `prd_iteration_mode` | 输出新一版 PRD 和版本记录。 |
| 业务材料充分，或已有 PRD / BRD 且用户要求“整理需求 / 标准化 PRD / 改写 PRD” | `prd_mode` | 输出完整场景中心 PRD，并附质量摘要和待确认问题。 |
| 已有 PRD / BRD 且用户明确要求“评估 / 评审 / 检查 / 是否可进入 BKN_Creator” | `review_mode` | 输出质量评分、缺口、改写建议和交接风险。 |
| PRD 已确认 | `handoff_mode` | 输出 BKN_Creator 交接摘要。 |

## 已有 PRD 的整理与评审分流

当用户提供一份已有 PRD / BRD / 需求文档时，不能只根据文件类型默认评审。必须结合用户动词判断主交付：

- 用户说“整理需求、整理 PRD、标准化、改写、输出正式文档、转成标准 PRD”时，主交付是新版标准 PRD，使用 `prd_mode`。
- 用户说“评估、评审、检查、打分、看看能不能进入 BKN_Creator”时，主交付是评审报告，使用 `review_mode`。
- 如果用户同时要求“先评审再整理”，先输出轻量评审摘要，再生成新版 PRD。
- 如果已有 PRD 信息不足以生成可评审 PRD，输出验证性结果或评审报告，并明确说明为什么不能生成新版 PRD。

`prd_mode` 可以包含质量摘要，但不能只输出评审；`review_mode` 可以包含改写建议，但不默认生成新版 PRD。

## 主线方法

不要从 BKN 元素开始问。先还原业务工作，再把候选建模项放到末尾交接。

```text
业务目标
→ 业务用户
→ 业务场景
→ 当前流程 / 目标流程
→ 输入 / 输出
→ 业务规则 / 判断标准
→ 系统 / 表单 / 数据
→ 权限 / 审批 / 异常
→ 界面 / 交互期望
→ 业务验收用例
→ BKN_Creator 交接摘要
```

如果处于 PRD 迭代阶段，先执行：

```text
调研大纲 / 本轮目标
→ 会议纪要及相关材料
→ 上一版 PRD
→ 本轮目标对齐
→ 新增事实 / 修正事实 / 废弃内容
→ 影响章节
→ 新一版 PRD
→ 版本记录与下一轮追问
```

## 场景拆解

每个场景至少记录：

- 场景名称；
- 主要用户；
- 触发方式；
- 当前流程；
- 目标流程；
- 业务输入；
- 业务输出；
- 业务规则；
- 异常边界；
- 人工确认点；
- 界面 / 交互期望；
- 业务验收标准。
- 该场景提升的业务能力；
- 关键决策点；
- 操作闭环。

每个 P0/P1 场景还必须完成一次正文四层业务收敛，但正文先用业务语言表达，而不是直接抛出建模术语；同时记录可由 Skill / Agent 支撑的业务任务线索，供第 15 节五层交接摘要使用。

| 业务标题 | 对应层级 | 必须回答的问题 |
|---|---|---|
| 需要识别的业务对象 | 概念模型层 | 为了让这个场景成立，业务上必须分清哪些可被管理、追踪或承担责任的实体、单据、计划、承诺、事件或事实载体？状态、结果、数量和口径先作为属性、枚举、派生结果或逻辑输出判定，不直接对象化。 |
| 需要表达的业务联系 | 关系层 | 这些对象之间存在什么消耗、满足、替代、归属、依赖、审批或产生关系？ |
| 需要判断、计算或推进的业务逻辑 | 动力层 | 场景中必须做出哪些判断、计算、排序、状态变化、提醒或草案生成？ |
| 需要控制的责任、权限与留痕 | 治理层 | 哪些人能看、能改、能确认、能导出，哪些动作必须留痕、审批或拒绝自动化？ |

输出时必须说明“为什么需要这一层”，并优先绑定场景目标、业务规则或验收用例。只列“对象候选 / 关系候选 / 动作候选”的名词清单，视为未完成场景收敛。

Agent / Skill 任务线索至少说明：帮谁、位于哪个流程节点、输出什么、不能自动做什么。它不是正文第五层建模表，而是为了避免第 15 节 Skill / Agent 应用层末尾补造。

### 本体论判别纪律

候选项完成分类后，必须继续通过 `references/bkn-requirement-ontology-discovery.md` 的本体建模门禁。分类决定“它是什么”，建模门禁继续判断“它为什么成立、边界在哪里、如何参与关系、逻辑、行动与治理”。

正文四层业务收敛前，必须先做“分类判定”，避免把所有业务名词都当对象。

| 分类 | 优先判定 |
|---|---|
| 对象候选 | 现实世界中可独立识别、可被管理、可参与多个场景或决策，且有稳定身份、生命周期或业务边界。 |
| 属性候选 | 某个对象的编码、状态、数值、描述或特征。 |
| 枚举候选 | 有限集合中的分类值、状态值、原因码；默认不对象化。 |
| 关系候选 | 两个对象之间稳定、可复用、具有业务含义的联系。 |
| 结果候选 | 一次计算、分析或视图输出；默认先视为派生属性、函数输出或分析结果。 |

判定规则：

- 业务名词不等于对象；
- `SKU 编码`、`库存状态`、`不可用原因类型` 这类内容优先怀疑为属性或枚举；
- `可用量`、`理论可生产量`、`缺口数量` 这类内容优先怀疑为派生结果；
- 如果说不清“它被谁管理、如何识别、何时变化、为何要单独存在”，不得直接升格为对象；
- 如果某项是否对象化尚不确定，写为“待下游建模判定”，不要用含混术语掩盖不确定性。

动力层也必须分类：

| 类型 | 判定 |
|---|---|
| 计算逻辑 | 只生成指标、数量、解释，不直接改变业务状态。 |
| 决策逻辑 | 根据规则输出满足、缺口、风险、优先级、通过 / 拒绝等结论。 |
| 状态变化 | 推动某个业务对象进入新的生命周期状态。 |
| 行动候选 | 用户或 Agent 发起一次受控业务变更，会修改对象、属性、关系或触发副作用。 |

治理层只保留真正的控制问题：谁能看、谁能改、谁能确认、哪些动作必须审批、哪些结果必须留痕、哪些自动化必须被拒绝、失败后如何处理。`查看`、`刷新`、`导出` 默认只是交互或系统能力，只有涉及权限、频率限制、敏感数据或审计时，才进入治理层。



### 交接摘要双门禁

输出 BKN_Creator 交接摘要前必须同时通过两道门禁：

#### 确认状态门禁

`业务已确认内容` 只接收已经被场景、规则、验收用例或原始材料明确支撑的事实。凡含以下信号之一，不得进入 `业务已确认内容`：

- 待确认、未确认、待对齐；
- 是否、范围待定、范围待确认；
- 候选、建议、可能；
- 字段待确认、容差未确认、是否自动下单未确认等括号补充。

处理方式：把一句混合表述拆开。已确认的业务原则留在 `业务已确认内容`；未确认的口径、字段、容差、范围、执行方式移入 `仍属建模候选` 或 `需下游建模阶段判定的问题`。

#### 本体一致性门禁

- 关系层出现的核心名词，必须先在概念模型层逐项说明“是什么 / 为什么需要”；
- 动力层输出的结果，必须判定为业务对象、派生结果或状态；
- `满足方案行`、`缺口行`、`报表行`、`看板指标`、`分析结果` 默认是结果候选或输出明细，不默认是对象；
- 只有当结果类内容具备稳定身份、生命周期、责任人或后续业务动作时，才可作为对象候选，并必须说明理由；
- 不能让关系层或动力层引入概念层未交代清楚的核心对象。

### 场景契约

每个 P0/P1 场景都必须形成最小契约：

| 契约项 | 要求 |
|---|---|
| 业务能力 | 说明该场景提升哪项工作能力，以及预期业务结果。 |
| 决策点 | 说明谁在何时做什么判断，依赖哪些证据，哪些是硬规则，哪些依赖专家判断，错判代价是什么。 |
| 操作闭环 | 说明发现、判断、建议、确认、执行、状态变化、追踪、反馈如何闭合。 |
| 必需事实 | 说明完成判断必须掌握哪些事实。 |
| 必需关系 | 说明完成判断必须表达哪些业务关系。 |
| 必需逻辑 | 说明完成判断必须有哪些计算、决策、状态变化或行动候选；行动候选必须另行说明触发、前置、影响对象和失败处理。 |
| 治理约束 | 说明权限、审批、留痕、拒绝自动化和失败处理。 |
| Agent / Skill 线索 | 说明 AI 帮谁、在哪个流程节点介入、输出什么、哪些内容不能自动执行。 |
| 验收映射 | 说明用哪些验收用例证明该能力成立。 |

如果某个“场景”只有页面和报表，没有清晰决策点、没有操作闭环、没有可衡量的业务结果，应降级为分析支持或候选能力，而不是直接作为核心 use case。

如果材料里只有“对象、关系、指标、Action”，需要反向追问：这些元素服务哪个业务场景、谁使用、在哪个流程节点使用、怎样验收。

## 业务规则

业务规则优先使用业务语言表达，例如：

- 可用库存如何扣减；
- 风险等级如何判断；
- 供应关系方向如何解释；
- 哪些动作只能生成草案；
- 哪些情况必须拒绝或转人工审批；
- 哪些数据缺失时不能给确定性结论。

不要在 PRD 主体中把业务规则写成 `operator` 或执行工厂算子。若后续确实需要，应放入 `BKN_Creator` 交接摘要的 `candidate_logic_properties`。

## 系统与数据发现

必须记录：

- 涉及的业务系统、表单、报表、Excel、API、数据库或数据视图；
- 是否为事实来源；
- 数据负责人；
- 关键字段或业务信息；
- 对象 ID 和跨系统映射问题；
- 数据质量风险；
- 延迟和刷新频率；
- 访问限制；
- 外部写回系统。

不要让表结构决定 BKN 对象模型。数据材料用于验证或实现业务对象。

## 场景化追问清单

PRD 正文中的待确认问题必须按业务场景组织。不要把业务问题、产品范围问题、系统数据问题和需下游建模阶段判定的问题混成一张表。

每个场景的追问优先使用业务语言：

| 追问类型 | 业务化追问方向 |
|---|---|
| 业务目标 | 这个场景解决谁的什么问题，如何判断业务结果变好？ |
| 当前流程 | 现在谁在什么系统、表单或 Excel 中完成这件事？ |
| 目标流程 | 系统介入后，人和系统如何分工，结果如何生效？ |
| 规则口径 | 判断满足、缺口、风险、优先级或通过/拒绝的业务标准是什么？ |
| 异常边界 | 数据缺失、冲突、超期、权限不足或结果不可信时如何处理？ |
| 人工确认 | 哪些结果只能作为建议或草案，不能自动生效？由谁确认？ |
| 验收样例 | 请给出典型、边界、失败或需人工确认的真实案例。 |

技术化问题必须转译。例如：

| 不推荐问法 | 推荐问法 |
|---|---|
| S3 是否写回 ERP、写回粒度？ | 采购建议生成后，业务希望它只是用于查看和导出，还是进入现有采购流程形成待办或草稿？正式生效前由谁确认？ |
| BOM 替代料是否本期必做？ | 当主料不足但存在可替代物料时，业务是否希望本期系统给出替代建议？如果本期不支持，会影响哪些典型判断？ |
| 需要哪些字段？ | 业务人员做判断时必须看到哪些信息？这些信息现在来自哪个系统或表单？ |

允许在 §13 末尾增加“产品范围确认问题”“系统与数据确认问题”，但它们不得替代每个场景的业务追问；建模边界、关系表达、指标 / 算子 / 逻辑能力挂载和行动执行模式等问题应进入交接摘要的 `需下游建模阶段判定的问题`。

## PRD 版本与迭代

PRD 是版本化资产。每次会议纪要、客户补充材料或业务确认进入 PRD 时，必须记录：

- 本轮输入来源；
- 本轮输入源清单和归档路径；
- 本轮调研目标是否被确认；
- 新增已确认事实；
- 对上一版 PRD 的修正事实；
- 废弃或移出范围的内容；
- 影响的 PRD 章节和场景；
- 版本号建议；
- 下一轮待确认问题。

`prd_iteration_mode` 标准输入：

| 输入 | 作用 |
|---|---|
| `research_outline` | 本轮调研大纲和原计划确认的问题。 |
| `meeting_notes_and_materials` | 会议纪要、录音转写、截图、表单、样例数据和客户补充材料。 |
| `previous_prd` | 上一版 PRD，作为本轮增删改基线。 |

`prd_iteration_mode` 标准输出：

| 输出 | 说明 |
|---|---|
| `updated_prd` | 新一版 PRD。 |
| `revision_summary` | 新增、修正、废弃和影响章节摘要。 |
| `change_log` | 版本记录。 |
| `unresolved_questions` | 待确认问题。 |
| `next_research_focus` | 下一轮调研重点。 |

## BKN_Creator 交接

PRD 末尾输出 `bkn_creator_handoff`。交接摘要必须遵循“场景先行、固定五层交接表、全局归并”的顺序，并始终使用中文业务语言。

需求发现阶段不输出机器可读 schema。它的职责是把业务讲清楚，而不是提前把 PRD 写成建模输入文件。

```text
业务场景
→ 场景内流程 / 规则 / 验收
→ 场景正文中的业务对象 / 业务联系 / 业务逻辑 / 治理控制收敛，以及 Agent / Skill 任务线索
→ 场景绑定的概念模型层、关系层、动力层、治理层、Skill / Agent 应用层候选
→ 全局归并为：业务已确认内容 / 仍属建模候选 / 需下游建模阶段判定的问题
```

中文业务可读摘要推荐结构。第 15 节必须按每个已承诺 P0 / P1 场景展开固定五层表格，不能写成 `对象：A、B；关系：A-B；逻辑：X` 这类压缩摘要；它必须比正文场景收敛更完整，不能更抽象。

```markdown
### 按场景收敛摘要

#### S1 <场景名称>

##### 概念模型层：需要识别的业务对象

| 业务对象 | 是什么 | 为什么需要 | 边界 | 确认状态 | 来源 / 验收映射 |
|---|---|---|---|---|---|
|  |  |  |  | 已确认 / 建模候选 / 待下游判定 |  |

对象说明必须体现身份线索、生命周期、责任角色、来源证据或管理动作；缺失时标为候选或待下游建模判定。

##### 关系层：需要表达的业务联系

| 业务联系 | 连接什么 | 业务含义 | 为什么需要 | 确认状态 | 来源 / 验收映射 |
|---|---|---|---|---|---|
|  |  |  |  | 已确认 / 建模候选 / 待下游判定 |  |

关系说明必须能还原方向、约束、有效期或适用范围；如果一端只是计算结果、诊断结果或报表行，优先放回动力层输出或结果候选，不进入关系层。

##### 动力层：需要判断、计算或推进的业务逻辑

| 类型 | 业务逻辑 | 怎么做 / 怎么判断 | 输出什么 | 边界 | 确认状态 | 来源 / 验收映射 |
|---|---|---|---|---|---|---|
| 指标 / 算子 / 状态变化 / 行动候选 |  |  |  |  | 已确认 / 建模候选 / 待下游判定 |  |

动力层说明必须区分指标、算子、状态变化和行动候选；行动候选还必须说明触发、前置条件、影响对象和失败处理。

##### 治理层：需要控制的责任、权限与留痕

| 治理点 | 控制什么 | 谁负责 / 谁使用 | 为什么需要 | 确认状态 | 来源 / 验收映射 |
|---|---|---|---|---|---|
|  |  |  |  | 已确认 / 建模候选 / 待下游判定 |  |

治理层说明必须体现责任、权限、审批、留痕、拒绝自动化或人工接管边界；普通查看、刷新、导出不默认进入治理层。

##### Skill / Agent 应用层：可支撑的业务任务

| Agent 任务 | 帮谁 | 做什么 | 输出要求 | 不能自动做什么 / 风险边界 | 确认状态 | 来源 / 验收映射 |
|---|---|---|---|---|---|---|
|  |  |  |  |  | 已确认 / 建模候选 / 待下游判定 |  |

在交接摘要中，如果某项更接近属性、枚举、派生结果、计算逻辑或交互能力，必须显式写出，不得为了填满五层而强行对象化、关系化、行动化或治理化。

### 全局归并摘要

#### 业务已确认内容
#### 仍属建模候选
#### 需下游建模阶段判定的问题
```


进入“业务已确认内容”必须同时满足：

- 来自至少一个明确场景；
- 有业务证据或验收用例；
- 不是工程推断；
- 业务含义清楚；
- 未标注“范围待确认”。

范围待确认的内容必须拆分。例如“替代料逻辑（范围待确认）”不能整体进入“业务已确认内容”，只能拆成：

- “业务已确认内容”：业务存在主料 / 替代料判断；
- “仍属建模候选”：是否本期实现替代料自动判断；
- “需下游建模阶段判定的问题”：替代关系如何建模。

PRD 正文只使用中文标题，保持业务可读性。

## 候选建模线索

这些线索只作为后台检查和 handoff 候选，不应喧宾夺主：

- 业务名词：进入分类判定入口，不能直接视为对象；
- “属于、包含、供应、影响、依赖、拥有”：候选关系；
- “评分、优先级、风险、影响范围、排序、推荐、模拟”：指标 / 算子 / 逻辑能力候选；
- “创建、提交、审批、关闭、发布、提醒、写回”：行动候选；
- “必须审批、不能自动、只能草案、需要审计”：治理要求；
- “问答、分析、建议、解释、总结”：候选 Skill / Agent 能力。
