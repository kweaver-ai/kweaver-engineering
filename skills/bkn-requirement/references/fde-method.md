# FDE 业务访谈方法

当需要解释或应用 FDE-style 需求发现方法时读取本文件。本文件只用于业务访谈、场景拆解和 PRD 表达，不用于在 PRD 主体中输出对象、关系、指标、算子或行动定义。

如需要把已确认 PRD 映射为 BKN_Creator 交接线索，再读取 `references/fde-bkn-mapping.md`。

## 核心原则

FDE-style 需求发现从用户需要获得的新业务能力开始，而不是从源系统、数据表、页面、模型技术或本体元素开始。

关键问题：

- 谁的工作会因为这个能力而改善？
- 哪个决策、流程或业务行动会变得更好？
- 用户现在如何完成这件事？
- 用户期望系统给出什么输入、输出、解释和下一步？
- 如何衡量成功？
- 哪些规则、权限、审批和异常边界不能被绕过？

## 访谈节奏

业务访谈中按这个顺序追问。

### 1. 业务结果

- “这个能力上线后，什么业务结果会改变？”
- “谁受益？谁负责结果？”
- “怎么衡量做成了？”
- “哪些内容明确不在本期范围内？”

### 2. 当前流程

- “流程从什么事件开始？”
- “谁处理每个步骤？”
- “每一步会看哪些系统、表单或报表？”
- “哪里会延迟、出错、重复处理或升级？”

### 3. 目标流程

- “理想情况下，用户希望系统帮他完成哪几步？”
- “哪些环节可以自动分析，哪些环节必须人工确认？”
- “输出之后，用户会做什么业务动作？”

### 4. 决策输入

- “做判断前会看哪些信息？”
- “哪些输入是必须的？”
- “哪些信息可信，哪些信息经常有争议？”
- “如果数据缺失或冲突，用户希望系统怎么提示？”

### 5. 业务规则

- “优先级、风险、影响范围、可用量、评分或推荐是怎么判断的？”
- “规则由谁维护？”
- “规则会不会按客户、产品、区域、时间或角色变化？”
- “哪些情况必须拒绝自动处理或转人工审批？”

### 6. 界面与体验

- “用户是在收件箱、详情页、分析工作台、聊天窗口、看板还是流程应用中工作？”
- “用户采取行动前必须看到什么？”
- “是否需要筛选、排序、钻取、导出、通知或写回？”

### 7. 治理

- “谁能看、谁能建议、谁能确认、谁能审批、谁能审计？”
- “哪些动作绝不能自动执行？”
- “出错时如何回滚或补偿？”

### 8. 验收

- “给几个典型案例、边界案例，以及 AI 必须拒绝或请求确认的案例。”
- “谁来确认标准答案？”
- “通过标准是什么？”

## 运营告警分诊案例

原始功能需求：

```text
路线运营分析师查看负责路线的告警收件箱，
根据优先级、航班详情、路线详情和组织影响进行分诊，
并对每个告警重新分派、解决或升级。
```

FDE 业务拆解：

| 需求短语 | 业务 PRD 表达 |
|---|---|
| 路线运营分析师 | 用户角色、职责和权限边界 |
| 告警收件箱 | 主要工作入口和界面期望 |
| 负责路线 | 用户职责范围和筛选条件 |
| 分诊告警 | 核心业务场景 |
| 优先级 | 业务判断标准 |
| 航班与路线详情 | 决策输入和证据 |
| 组织影响 | 业务影响解释 |
| 重新分派、解决、升级 | 用户可采取的业务动作和审批边界 |

访谈展开：

```text
FDE：分析师每天第一步做什么？
业务：打开告警收件箱。
=> PRD：工作入口是运营告警收件箱。

FDE：告警只是一条通知，还是一个需要跟踪状态的工作项？
业务：是工作项，可以分派、调查、升级、解决。
=> PRD：告警分诊场景需要状态、处理人和处理记录。

FDE：分析师怎么判断先处理哪条？
业务：看严重度、航班影响、路线重要性、组织影响。
=> PRD：优先级判断需要展示影响证据和排序依据。

FDE：分析师可以做哪些处理？
业务：重新分派、解决、升级。
=> PRD：处理动作必须有权限、必填原因和审计记录。

FDE：怎么判断系统做得好？
业务：高优先级告警不能漏，平均处理时间下降，误升级率不能太高。
=> PRD：业务验收指标和测试用例。
```

## KWeaver 协作链

KWeaver 中对应的 FDE-style 协作链：

```text
自然语言 / 录音转写 / 会议纪要 / PRD
  -> bkn-requirement 生成业务场景 PRD
  -> BKN_Creator 交接摘要
  -> bkn-creator 建模、绑定、测试
  -> kweaver-core 平台查询和执行
  -> TraceAI / Eval / 业务反馈
```

`bkn-requirement` 负责分析和结构化需求，不直接修改平台资产。
