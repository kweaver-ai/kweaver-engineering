# FDE 到 KWeaver BKN 的后置映射

仅在以下情况读取本文件：

- PRD 已基本形成，需要生成或检查 `BKN_Creator` 交接摘要；
- 用户明确要求解释 FDE-style 需求拆解与 KWeaver BKN 的对应关系；
- `handoff_mode` 需要把业务场景转成候选建模线索。

不要在业务 PRD 主体中直接输出本文件的建模字段。

## 需求组件映射

| 需求组件 | 含义 | KWeaver BKN 交接线索 |
|---|---|---|
| Object model | 核心对象、派生对象、用例对象和 Link | 概念模型层、关系层候选 |
| Lifecycle diagram | 用户通过 Actions 修改对象状态的状态机 | 状态变化、行动候选、审批和失败处理 |
| Enrichments | 支撑决策输入的数据增强 | 动力层候选，可能是指标、算子或解释逻辑 |
| Interface expectations | 不同界面的意图和交互期望 | DIP / Agent 工作台 / Skill 交互需求 |
| Governance | 权限、安全、审批和审计 | 治理层候选，包含责任、权限、审批、留痕和拒绝自动化边界 |

## 从业务 PRD 提取 handoff 线索

| PRD 内容 | handoff 候选 |
|---|---|
| 业务名词、工作项、事件、任务、提案、证据 | 进入分类判定入口；只有具备稳定身份、生命周期、责任或管理动作时才成为对象候选 |
| 属于、包含、依赖、影响、供应、负责、拥有 | 关系候选；必须连接两个已说明对象并表达业务含义 |
| 评分、排序、优先级、可用量、影响范围、推荐、模拟 | 动力层候选；需区分指标、算子、派生结果或解释逻辑 |
| 创建、提交、审批、关闭、发布、提醒、写回 | 状态变化或行动候选；必须说明触发、前置、影响对象、失败处理和人工确认 |
| 必须审批、不能自动、只读、脱敏、回滚 | 候选治理要求 |
| 问答、分析、建议、解释、生成草案 | 候选 Skill / Agent 能力 |

## 交接分层

`bkn_creator_handoff` 必须先按场景展开固定五层交接表，再做全局归并，并始终使用业务语言。不得只输出全局三段摘要。

按场景五层交接表：

| 层级 | 必须说明 |
|---|---|
| 概念模型层 | 业务对象是什么、为什么需要、身份 / 生命周期 / 责任 / 边界是什么。 |
| 关系层 | 连接哪些对象、业务含义、方向 / 约束 / 有效范围是什么。 |
| 动力层 | 指标、算子、状态变化或行动候选分别是什么，输入、规则、输出和边界是什么。 |
| 治理层 | 谁负责、谁确认、哪些权限 / 审批 / 留痕 / 拒绝自动化 / 失败处理需要控制。 |
| Skill / Agent 应用层 | 帮谁、在哪个流程节点、做什么、输出什么、不能自动做什么。 |

全局归并摘要：

| 分组 | 内容 |
|---|---|
| 业务已确认内容 | 业务专家可直接确认，且可追溯到场景、规则、验收用例或输入材料的内容。 |
| 仍属建模候选 | AI 工程师根据 PRD 推导出的对象、关系、指标、算子、行动、治理或 Skill / Agent 候选。 |
| 需下游建模阶段判定的问题 | 需要下游决定的对象边界、关系表达、指标 / 算子 / 逻辑能力挂载、行动执行模式、治理策略等问题。 |

## 注意事项

- `operator` 不是业务逻辑本身，而是对象或对象集合上的逻辑能力候选。
- PRD 主体应说“优先级如何判断”“可用量如何计算”“影响范围如何解释”，不要提前说“调用某个 operator”。
- 高风险业务动作优先建成草案、建议或审批流程，不应默认自动执行。
- 数据视图、字段映射和接口绑定属于下游建模或实现线索，不进入 `bkn-requirement` 的已确认业务内容或建模候选主表。
