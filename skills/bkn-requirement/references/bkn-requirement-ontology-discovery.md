# BKN 需求发现本体识别方法

## 定位

本文件用于 `bkn-requirement` 的需求发现阶段，在生成场景收敛小结和 `BKN_Creator` 交接摘要前使用。

它不是完整 BKN 设计规格，也不是字段命名规范；它定义的是：如何把业务材料中的人、事、物、规则、判断、动作和责任边界，整理成后续 BKN 建模可接住的本体候选。

本文件是 `references/bkn-methodology.md` 的需求发现阶段裁剪版。`bkn-methodology.md` 负责统一对象、事实属性、关系、指标、算子、行动、治理和 Agent / Skill 应用层边界；本文件只负责把这些边界转译成 PRD 场景收敛、追问和交接摘要可使用的业务表达。

仓库内的 `../../common/bkn-methodology.md` 是公共源方法论；`references/bkn-methodology.md` 是随 `bkn-requirement` 单 Skill 分发安装的快照，用于保证 `npx skills add ... --skill bkn-requirement` 后引用完整。

本 skill 输出仍以中文业务语言为主，不把 PRD 写成平台模型文件。

## 总原则：先还原业务世界，再抽象本体结构

不要从字段、页面、报表、算法输出开始建模。必须先回答：业务上到底在管理什么、判断什么、改变什么、由谁负责、留下什么证据。

每个核心场景至少完成六类判定：

| 层级 | 要回答的问题 | 输出形态 |
|---|---|---|
| 对象 | 业务上有哪些可独立识别、管理、追踪、负责的实体或事实载体？ | 对象候选 |
| 属性 / 枚举 | 哪些只是描述对象的编码、状态、数量、日期、类型、原因或口径？ | 属性候选 / 枚举候选 |
| 关系 | 哪些对象之间存在稳定、可复用、有业务含义的联系？ | 关系候选 |
| 逻辑 | 场景中必须完成哪些指标读取、计算、判断、排序、推荐、解释或状态推进？ | 指标候选 / 算子候选 / 计算逻辑 / 决策逻辑 |
| 行动 | 哪些动作会改变对象、影响对象、影响外部系统、触发写回或进入审批？ | 行动候选 |
| 治理 | 谁能看、谁能改、谁能确认、什么必须审批、留痕或拒绝自动化？ | 治理要求 |

## 分类判定输出样式

在进入正文四层小结或第 15 节五层交接前，先输出或内部完成分类判定。分类判定不是技术建模表，而是防止把字段、结果、角色、规则直接升格为对象的业务检查。

| 类型 | 候选内容 | 判断依据 | 处理方式 |
|---|---|---|---|
| 对象候选 |  | 有稳定身份、生命周期、责任、关系或管理动作 | 进入概念模型层 |
| 属性 / 枚举候选 |  | 描述对象的编码、数量、状态、类型、日期或等级 | 写入对象边界或待下游建模判定 |
| 关系候选 |  | 两个已说明对象之间存在稳定业务含义 | 进入关系层 |
| 结果候选 |  | 来自判断、计算、推荐、诊断或报表输出 | 默认进入动力层输出，不直接对象化 |
| 指标候选 |  | 有统一口径、对象归属和业务用途 | 进入动力层 |
| 算子 / 逻辑能力候选 |  | 需要读取输入、按规则判断、计算、排序、解释或推荐 | 进入动力层 |
| 状态变化 / 行动候选 |  | 会改变对象、影响对象或触发外部系统 | 进入动力层行动候选或治理层，并补触发、前置、影响和失败处理 |
| 治理要求 |  | 涉及责任、权限、审批、留痕、敏感数据、拒绝自动化或人工接管 | 进入治理层 |

## 1. 对象建模规范

### 1.1 什么可以成为对象

对象应是业务世界中的稳定实体、单据、计划、承诺、事件、记录或可被追踪的事实载体。对象必须能被业务用户理解，具备稳定识别方式，并能够承载事实属性、关系、指标、算子、行动或治理。规则、口径、视图、角色名、建议结果、算法输出不应默认对象化。

一个候选项只有满足以下至少两项，才适合进入对象候选：

- 有稳定身份，可以被业务反复引用；
- 有生命周期或状态变化；
- 有责任人、归属方、管理部门或审批人；
- 会参与多个业务关系；
- 会被确认、调整、追踪、写回、审计或复盘；
- 离开它，场景中的关键判断无法解释清楚。

反向门禁：如果一个候选项更像规则、口径、角色、视图、指标、结果、建议或临时分类，即使业务上重要，也不得直接作为对象。应先归入事实属性、枚举、关系约束、指标、算子输出、行动草案或治理要求。

### 1.2 什么不应直接成为对象

以下内容默认不对象化，必须先降级判定：

- 编码：如 `SKU`，优先是 `产品.sku编码`；
- 数值 / 指标：如 `库存余额`、`可用库存`、`缺口数量`；
- 状态 / 类型：如 `库存状态`、`风险等级`；
- 统计视图：如 `分仓分状态库存余额`；
- 报表行 / 明细行：如 `满足方案行`、`缺口行`；
- 页面组件：如 `查询结果`、`分析报表`、`看板指标`；
- 算法输出：如 `推荐结果`、`排序结果`，除非它有后续确认、审批、追踪或写回；
- 业务规则 / 口径：如 `替代规则`、`分配规则`、`可用量口径`，默认作为关系约束、指标口径、算子输入或治理要求；
- 场景角色名：如 `拆料来源产品`、`候选物料`、`优先需求`，默认是已有对象在场景中的角色，不新建对象。

### 1.3 对象表达规范

对象表达必须包含三句话：

| 必填项 | 要求 |
|---|---|
| 是什么 | 用业务语言说明它代表的实体或事实载体。 |
| 为什么需要 | 说明它支撑哪个业务判断、关系、动作或验收目标。 |
| 边界 | 说明它不包含什么，尤其要排除字段、指标、报表行或临时结果。 |

示例：

| 不合格表达 | 问题 | 合格表达 |
|---|---|---|
| `产品（SKU）` | 对象与编码混杂 | `产品` 是被需求、库存、BOM 和交付判断共同引用的业务实体；`sku编码` 是产品的识别属性，不是对象名的一部分。 |
| `库存余额（分仓分状态）` | 把统计口径当对象 | 应先抽象为 `库存` 对象；仓库、批次、库存状态、库存数量、可用数量、占用数量优先作为属性、枚举、关系或计算结果。只有业务按批次独立冻结、释放、盘点、追踪或审计时，才把 `库存批次` 作为对象候选。 |
| `满足方案行` | 报表行对象化 | 若只是输出行，应归为履约建议生成的结果明细；只有当建议会被确认、调整、追踪、失效时，才可抽象为 `履约建议`。 |



### 1.4 规则、角色与事件的对象化边界

需求发现阶段必须区分真实业务对象、业务关系上的约束、业务逻辑和场景角色。

| 候选项 | 默认处理 | 升格为对象的条件 |
|---|---|---|
| 业务规则 | 关系约束 / 指标口径 / 算子输入 / 治理要求 | 只有规则本身有编号、版本、生效期、审批、发布、废止、责任人和审计时，才可建为规则类对象。 |
| BOM / 替代规则 | `产品 消耗 物料`、`物料 替代 物料` 关系上的约束，或物料展开 / 替代覆盖逻辑 | 只有企业把 BOM 或替代规则作为主数据独立维护，具备编号、版本、生效期、审批、发布废止和适用范围时，才作为对象候选。 |
| 场景角色名 | 已有对象在场景中的角色 | 不直接对象化；例如 `拆料来源产品` 应回到 `产品` 或 `库存`，角色写在关系、逻辑或说明中。 |
| 建议 / 方案 / 诊断 | 逻辑输出 / 结果候选 | 只有会被确认、调整、分派、关闭、复盘或审计时，才升格为对象。 |
| 业务事件 | 对象候选 | 如果事件有发生时间、影响对象、来源、状态和处置动作，可以作为对象。 |

判断口诀：

```text
实体 / 单据 / 事件 → 可能是对象；
规则 / 口径 / 算法 → 优先是逻辑；
A 对 B 的约束 → 优先是关系属性或关系约束；
某对象在某场景里的叫法 → 优先是角色，不新建对象。
```

### 1.5 对象升格约束：先核心对象，后细分对象

需求发现阶段不要把同一业务对象的粒度、状态、数量、口径直接拆成多个对象。应先识别核心业务对象，再判断是否需要细分对象。

以库存为例：

| 候选项 | 默认分类 | 升格为对象的条件 |
|---|---|---|
| 库存 | 对象候选 | 表达某产品或物料当前可被管理和判断的存量事实。 |
| 库存批次 | 属性 / 细分口径候选 | 只有业务按批次独立追踪、冻结、释放、盘点、审计或形成关系时，才升格为对象。 |
| 库存状态 | 属性 / 枚举 | 默认不对象化。 |
| 仓库 | 对象候选 | 仓库有管理责任、用途和权限边界，可作为对象；库存与仓库形成归属关系。 |
| 库存数量 / 可用数量 / 占用数量 | 属性 / 计算结果 | 默认不对象化。 |
| 库存占用 | 扣减事实 / 属性 / 关系候选 | 只有占用需要独立释放、审批、追踪、失效或审计时，才升格为对象。 |

判断口诀：如果业务问的是“还有多少、在哪个仓、什么状态、被占多少”，通常是库存对象的属性、枚举或计算逻辑；如果业务会对它单独审批、释放、冻结、盘点、追责，才考虑拆对象。

## 2. 属性、枚举、规则与关系约束建模规范

属性用于描述对象，枚举用于表达有限取值。规则和口径用于约束关系或驱动逻辑。不要因为一个词在业务上重要，就把它对象化。

### 2.1 属性判定

以下内容优先作为属性：

- 编码：产品编码、sku编码、工单号、采购单号；
- 数量：库存数量、占用数量、缺口数量、可用数量；
- 日期：需求日期、交付截止日、预计到货日；
- 状态：库存状态、工单状态、建议状态；
- 类型 / 原因 / 等级：需求类型、不可用原因、风险等级；
- 口径：是否计入在制、是否计入在途、可替代范围。

### 2.2 枚举判定

当一个属性有有限业务取值，且取值集合本身不需要生命周期、责任或动作时，应作为枚举候选。

示例：

- `库存.库存状态`：可用、冻结、质检、报废；
- `履约评估.满足状态`：可满足、部分满足、不可满足；
- `缺口诊断.缺口原因类型`：库存不足、在途未达、BOM 缺失、替代料未确认。

## 3. 关系建模规范

关系不是字段关联，也不是 SQL join。关系必须表达两个对象之间稳定、可复用、有业务含义的联系。

### 3.1 关系成立条件

一条关系候选必须说明：

| 必填项 | 要求 |
|---|---|
| 连接什么 | 明确 A 对象与 B 对象。 |
| 业务含义 | 说明这种联系在业务上代表什么。 |
| 为什么需要 | 说明没有这条关系，哪个判断、逻辑或行动无法成立。 |
| 方向与约束 | 如有必要，说明方向、数量约束、时间有效性或范围。 |

### 3.2 关系类型参考

| 类型 | 示例 | 业务含义 |
|---|---|---|
| 归属 | `库存 存放于 仓库` | 说明库存事实发生在哪个管理地点。 |
| 需求 | `客户需求 需求 产品` | 说明需求指向哪个产品。 |
| 消耗 | `生产计划 消耗 物料` | 说明生产计划会占用哪些物料。 |
| 满足 | `客户需求 由 库存 满足` | 说明需求可由哪些库存事实支撑。 |
| 替代 | `物料 替代 物料` | 说明两个物料之间存在业务允许的替代关系；替代条件、生效期、审批要求优先作为关系约束或逻辑口径。 |
| 占用 | `生产工单 占用 库存` | 说明库存中有多少数量已被业务安排占用；默认是库存的扣减事实或属性，只有需要独立释放、审批、追踪时才升格为对象。 |
| 产生 | `评估任务 产生 处置建议` | 仅当产出结果具备确认、分派、关闭、复盘等生命周期时，才可作为关系另一端；普通缺口诊断、分析结论或报表行优先作为动力层输出或结果候选。 |

### 3.3 关系反例

以下表达不合格：

- `产品-库存余额关系`：余额不是对象；
- `满足方案行关联需求`：报表行不是对象；
- `缺口行关联SKU`：SKU 是编码，缺口行是结果行；
- `页面字段关联`：技术实现关系，不是业务关系。

## 4. 逻辑建模规范

逻辑用于说明场景如何判断、计算、排序、推荐、解释或推进状态。逻辑不等于对象，也不等于行动。

### 4.1 逻辑分类

| 类型 | 判定 | 示例 |
|---|---|---|
| 计算逻辑 | 生成数量、金额、日期、比例等结果，不直接改变业务事实。 | `库存可用量计算`、`理论可生产量计算` |
| 决策逻辑 | 根据规则给出可 / 不可、风险、优先级、原因等判断。 | `需求可满足性评估`、`替代料覆盖判断` |
| 排序 / 推荐逻辑 | 在多个候选方案中给出优先级或建议。 | `履约建议排序`、`供应风险分级` |
| 解释逻辑 | 说明结果为什么成立、证据来自哪里。 | `缺口原因解释`、`库存占用解释` |
| 状态逻辑 | 推动对象进入新状态，但未必由系统自动执行。 | `履约建议状态推进` |

### 4.2 逻辑表达规范

每条逻辑必须说明：

| 必填项 | 要求 |
|---|---|
| 怎么判断 / 计算 | 用业务规则描述，不写技术算子。 |
| 输入事实 | 说明依赖哪些对象、属性和关系。 |
| 输出结果 | 说明输出的是状态、数量、原因、建议、排序还是解释。 |
| 边界 | 说明不适用条件、人工确认点、数据不足时如何处理。 |

示例：

- `库存可用量计算`：根据库存数量、库存状态、仓库用途、批次信息、已占用数量和是否允许计入的业务口径，计算可用于承诺的数量；输出可用数量和不可用原因。
- `需求可满足性评估`：基于客户需求、产品、库存、在制 / 在途事实和替代规则，判断需求可满足、部分满足或不可满足，并输出原因。

## 5. 行动建模规范

行动是受控业务变化，不是普通按钮，也不是所有用户操作。

### 5.1 什么可以成为行动候选

一个动作只有满足以下任一条件，才进入行动候选：

- 会修改对象、事实、关系或状态；
- 会触发审批、写回、通知、锁定、释放、生成单据等副作用；
- 需要权限控制、人工确认、失败处理或审计；
- 其执行结果会影响后续业务判断。

### 5.2 什么不应成为行动候选

以下默认不是行动，只是交互或查询能力：

- 查看、搜索、筛选、刷新；
- 普通导出；
- 切换视图；
- 打开详情页。

如果这些动作涉及敏感数据、频率限制、审批、审计或外部写回，才进入治理或行动候选。

### 5.3 行动表达规范

每个行动候选必须说明：

| 必填项 | 要求 |
|---|---|
| 谁发起 | 用户、角色、Agent 或系统触发条件。 |
| 改变什么 | 修改哪个对象、事实、关系、状态或外部流程。 |
| 前置条件 | 什么情况下允许执行。 |
| 人工确认 | 是否需要确认、审批或复核。 |
| 失败处理 | 数据不足、权限不足、写回失败时如何处理。 |
| 留痕 | 记录什么证据，供谁追溯。 |

示例：

- `确认履约建议`：PMC 确认某条履约建议后，建议状态从待确认变为已确认，并记录确认人、时间、依据和例外说明。
- `释放库存占用`：在授权条件下释放某工单对库存的占用数量，影响后续可用量计算，必须留痕；是否需要独立的占用对象由业务生命周期决定。

## 6. 治理建模规范

治理用于表达责任、权限、审批、留痕、拒绝自动化和风险边界。

### 6.1 治理必须回答

| 问题 | 示例 |
|---|---|
| 谁能看 | 哪些角色可见敏感库存、客户需求或供应风险。 |
| 谁能改 | 谁能修改需求、确认建议、调整替代料。 |
| 谁能确认 | 哪些结论必须由 PMC、计划、采购或主管确认。 |
| 什么要审批 | 写回、锁库、替代、改单、释放占用。 |
| 什么要留痕 | 判断依据、人工调整、例外放行、失败原因。 |
| 什么不能自动化 | 高风险替代、跨客户挪料、超权限承诺。 |

### 6.2 治理反例

以下表达过浅：

- `查看权限`：没有说明看什么、谁看、为什么敏感；
- `导出治理`：没有说明导出内容、风险、审计要求；
- `刷新控制`：除非涉及频率限制或数据一致性，否则只是交互能力。

## 7. Agent / Skill 应用层规范

Agent 不是“会查表的工具”，而是面向角色完成业务任务的执行者。Agent 任务必须连接对象、关系、逻辑、行动和治理。

每个 Agent 任务必须说明：

| 必填项 | 要求 |
|---|---|
| 帮谁 | 明确业务角色。 |
| 做什么业务任务 | 不是“查询数据”，而是完成判断、解释、建议、追踪或草案生成。 |
| 依赖什么事实 | 说明对象、关系、规则或输入材料。 |
| 输出什么 | 建议、原因、风险、清单、草案、待确认问题。 |
| 受什么控制 | 权限、人工确认、审计、拒绝自动化边界。 |

示例：

- `需求履约预检 Agent`：帮助 PMC 在承诺交付前评估客户需求是否可满足，输出满足状态、关键缺口、建议动作和需要人工确认的问题。
- `缺口诊断 Agent`：帮助计划员解释不可满足原因，输出缺口物料、影响需求、在途 / 在制可覆盖情况和证据来源。

## 8. 端到端建模流程

每个 P0/P1 场景按以下顺序处理：

```text
业务目标
  ↓
关键决策点
  ↓
必须识别的业务对象
  ↓
对象属性 / 枚举边界
  ↓
对象之间的业务关系
  ↓
完成判断所需的业务逻辑
  ↓
会改变业务事实的行动
  ↓
责任、权限、审批、留痕和拒绝自动化边界
  ↓
Agent / Skill 可支撑的业务任务
  ↓
验收用例验证
```

不得跳过业务目标和关键决策点，直接从字段、报表或技术对象开始。

## 9. 重点反例改写

| 原表达 | 问题 | 正确处理 |
|---|---|---|
| `产品（SKU）` | 对象与编码混杂 | 对象写 `产品`；编码写 `产品.sku编码`；若业务上区分销售品项 / 库存品项，再按生命周期和管理边界拆对象。 |
| `库存余额（分仓分状态）` | 指标 / 视图对象化 | 先建模为 `库存`；仓库、批次、状态、数量、占用数量作为属性 / 枚举 / 关系 / 计算口径；余额作为计算结果或快照属性。 |
| `满足方案行` | 报表行对象化 | 默认是 `履约建议生成` 的输出明细；若可确认、调整、追踪、失效，才可抽象为 `履约建议` 对象。 |
| `缺口行` | 结果行对象化 | 默认是 `履约缺口诊断` 的输出明细；若缺口会被分派、关闭、复盘，才可抽象为 `缺口诊断` 或 `缺口事项`。 |
| `可用库存` | 派生指标对象化 | 表达为 `库存可用量计算` 的输出，或 `库存.可用数量`。 |
| `BOM 清单` | 把规则口径对象化 | 默认写为 `产品 消耗 物料` 的关系约束和物料展开逻辑输入；只有有 BOM 编号、版本、生效期、审批、发布废止生命周期时才对象化。 |
| `替代规则` | 把规则对象化 | 默认写为 `物料 替代 物料` 的关系约束，或 `替代料覆盖判断` 的逻辑输入；只有有版本、生效期、审批、发布废止生命周期时才对象化。 |
| `拆料来源产品` | 把场景角色对象化 | 回到 `产品` 或 `库存` 对象；“可拆用”是场景角色或逻辑判断结果，不新建对象。 |
| `确认方案` | 动作边界不清 | 若改变建议状态，写为行动候选 `确认履约建议`，并说明前置、权限、留痕和失败处理。 |
| `导出结果` | 普通交互动作 | 默认不进入行动候选；涉及敏感数据或审计时写入治理要求。 |

## 10. 输出门禁

进入 PRD 场景收敛小结或 `BKN_Creator` 交接摘要前，必须检查：

- 是否从业务目标和决策点出发，而不是从字段、页面、报表出发；
- 对象是否有身份、生命周期、责任、关系或管理动作；
- 编码、状态、数量、日期、类型、原因是否优先降级为属性或枚举；
- 关系是否连接两个已说明的对象，并具备业务含义；
- 指标或算子是否说明输入、规则、输出和边界；
- 行动是否真的改变业务事实或触发受控副作用；
- 治理是否说明责任、权限、审批、留痕或拒绝自动化边界；
- Agent 任务是否面向角色交付业务判断或行动建议，而不是只查表、生成报表；
- 结果类内容是否被默认写成对象；
- 所有候选是否能映射到场景目标、业务规则或验收用例。
