# 会议纪要与录音转写处理

当输入是录音转写、会议纪要、访谈摘要，或需要基于会议纪要更新 PRD 时读取本文件。

## 处理步骤

1. **识别会议元数据**
   - 会议日期；
   - 参会人和角色；
   - 业务领域；
   - 目标场景；
   - 提到的源材料；
   - 本轮 `source-manifest.md`，或需要登记 / 复制归档的输入文件；
   - 本轮调研大纲或原计划确认的问题；
   - 上一版 PRD 或当前文档版本。

2. **抽取已确认事实**
   - 业务专家明确认可的说法；
   - 当前流程步骤；
   - 当前系统、表单、报表或数据源；
   - 已确认业务规则、判断标准和人工确认点；
   - 明确的权限、审批或拒绝边界。

3. **抽取假设**
   - AI 工程师推断的内容；
   - 不清楚的业务术语；
   - 未验证的数据可用性；
   - 尚未确认的场景边界、规则口径、系统字段或验收样例。

4. **抽取冲突**
   - 同一概念多个名称；
   - 规则或指标口径不一致；
   - 负责人或事实来源说法不一致；
   - 权限或审批规则有争议。

5. **抽取业务术语**
   - 场景名；
   - 业务对象名；
   - 流程步骤；
   - 状态名；
   - 动作动词；
   - 规则、标准和指标；
   - 系统、表单和数据名。

6. **生成追问问题**
   - 围绕业务场景、流程、规则、数据、权限、界面和验收追问；
   - 对象、属性 / 枚举、关系、结果、指标 / 算子 / 逻辑能力、状态变化 / 行动候选和治理要求只作为 AI 工程师内部分类线索。

7. **生成 PRD 迭代输入**
   - 记录本轮输入来源和归档路径；
   - 对齐本轮调研目标：哪些问题已确认，哪些仍未确认；
   - 标注新增已确认事实；
   - 标注对上一版 PRD 的修正；
   - 标注废弃或移出范围的内容；
   - 标注影响的 PRD 章节；
   - 生成下一轮调研重点。

## 标注标签

在纪要处理中使用以下标签：

- `CONFIRMED`：业务或技术负责人已确认。
- `ASSUMPTION`：推断但未确认。
- `CONFLICT`：存在矛盾说法。
- `MISSING`：必需信息缺失。
- `RISK`：业务、数据、治理或交付风险。
- `FOLLOW_UP`：下次访谈需要追问的问题。
- `PRD_ADD`：应新增到 PRD 的内容。
- `PRD_UPDATE`：应修订上一版 PRD 的内容。
- `PRD_REMOVE`：应从 PRD 中废弃或移出范围的内容。

## 输出结构

```markdown
## 会议摘要

### 本轮输入来源
### 参会人
### 本轮调研目标对齐
### 已确认事实
### 假设
### 冲突
### 业务场景
### 业务规则与判断标准
### 提到的系统、表单与数据
### 权限、审批与风险
### 业务验收样例
### 对 PRD 的影响
### 下一轮调研重点
### 轻量交接线索（非完整第 15 节）

> 本节只记录会议纪要中的初步分类线索，不替代正式 PRD 或 handoff 的第 15 节。若需要输出完整交接摘要，必须按固定五层表格逐场景展开，并补确认状态与来源 / 验收映射。
### 追问问题
```

## PRD 迭代输出

当用户输入“调研大纲 + 会议纪要及材料 + 上一版 PRD”时，输出以下结构，供 `prd_iteration_mode` 使用：

```markdown
## 本轮调研更新摘要

| 项目 | 内容 |
|---|---|
| 本轮输入 |  |
| 输入源清单 |  |
| 上一版 PRD |  |
| 本轮原计划确认 |  |
| 实际已确认 |  |
| 仍未确认 |  |

### 新增已确认事实
### 修正事实
### 废弃 / 移出范围内容
### 影响的 PRD 章节
### 待确认问题
### 下一轮调研重点
```

不要把会议中未确认的 AI 工程师推断写成业务已确认事实。无法判断时放入“假设”或“待确认问题”。

## 识别经验

- 业务用户反复提到的名词，通常是业务对象线索。
- “先、再、如果、否则、超过、低于、必须”等表达，通常是业务规则线索。
- “提交、审批、关闭、发布、提醒、写回”等动词，通常是行动或草案线索。
- “必须审批”“只有经理”“不能自动”等说法，通常指向治理需求。
- 系统名、报表、Excel、API、数据库表等，通常指向数据发现需求。
- 用户问题和历史案例，应转成业务验收用例。
