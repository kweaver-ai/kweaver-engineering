# KWeaver BKN 方法论

## 目标

本文件是 KWeaver BKN 的方法论基座，供其他 Skill 引用。它不负责独立产出 PRD、`.bkn` 文件、数据绑定或平台操作，而是统一 Agent 在各类 BKN 相关任务中的本体判别、建模边界、治理边界和验证口径。

适用链路：

```text
业务材料理解
→ 本体候选判别
→ 概念模型建模
→ 动力层建模
→ 风险治理建模
→ Skill / Agent 开发设计
→ 验证与验收
```

它不是 `.bkn` 语法手册，也不是需求分析模板。它提供的是可被其他 Skill 复用的方法论规则。

## 方法论定位

BKN 本体建模的目标不是复刻数据库、接口或报表结构，而是构建一个面向 AI 理解、推理、行动和治理的业务知识网络。

BKN 应围绕以下核心问题展开：

- 业务中有哪些稳定对象；
- 对象之间有哪些业务关系；
- 对象有哪些基础事实属性；
- 哪些属性需要计算、推理或统一口径；
- 哪些业务动作可以被 AI 建议或执行；
- 哪些动作存在风险，需要治理；
- Skill 如何基于这些本体完成任务。

BKN 不是单纯的知识图谱，也不是单纯的数据模型。它同时表达业务对象、业务关系、数据属性、动态逻辑、可行动能力、风险治理和 Skill 消费方式。

## 引用协议

其他 Skill 引用本方法论时：

- 本文件只能作为公共方法论材料读取，不得作为独立 Skill 触发；
- 本文件不直接生成 PRD、`.bkn`、平台对象、数据绑定或执行操作；
- 引用方必须结合自身 `SKILL.md` 的阶段、输出契约和安全边界使用；
- 如与执行型 Skill 的流程细节冲突，以执行型 Skill 的阶段规则为准，但不得绕过本文的本体边界、行动边界和治理原则；
- 对外表达使用业务语言，不要求业务用户理解 BKN 术语；
- 先识别业务场景、决策点、对象、关系、逻辑、行动和治理，再形成建模建议；
- 只在用户明确要求生成 `.bkn`、绑定数据来源、推送网络或执行平台操作时，交由对应执行型 Skill 或平台工具处理；
- 对不确定内容标注“候选”或“待下游建模判定”，不得伪装成已确认事实。
- 必须区分已确认事实、建模候选、降级项、待业务确认和待下游建模判定。

最小引用检查：

1. 是否从业务目标和关键决策点开始，而不是从字段、页面、报表、接口或工具开始；
2. 是否区分对象、事实属性、关系、逻辑能力、行动和治理；
3. 是否把不确定内容标为候选、待业务确认或待下游建模判定；
4. 是否拒绝高风险自动执行，并说明审批、审计、回滚或人工确认边界；
5. 是否给出可验证的验收、证据链、留痕或 Eval 要求。需求发现阶段只表达业务证据链、验收和留痕要求；平台级 Trace / Eval 设计由下游 Agent、Action 或运行治理阶段处理。

## 一、端到端工作法

每个 P0/P1 场景按以下顺序处理：

```text
业务目标
→ 关键决策点
→ 业务对象候选
→ 属性 / 枚举 / 规则 / 结果降级判断
→ 对象关系
→ 判断、计算、推荐、解释逻辑
→ 会改变业务事实的行动
→ 责任、权限、审批、留痕和拒绝自动化边界
→ Skill / Agent 任务契约
→ 验收用例、证据链和下游 Trace / Eval 要求
```

不得从字段、页面、报表、接口或工具开始建模。

## 二、概念模型层方法

概念模型层回答：

```text
业务世界中有哪些对象？
它们如何被识别？
它们之间有什么稳定业务关系？
```

### 2.1 `concept_group`

`concept_group` 用于组织一组相关业务概念，帮助 Skill 理解业务边界和上下文。

使用原则：

- 按业务领域、业务能力或业务场景组织；
- 不按数据库、表、系统模块或技术分层简单划分；
- 分组应能帮助 Agent 判断对象、关系、逻辑和行动的业务语境。

示例方向：

```text
供应商管理
采购履约
风险事件
设备维护
客户服务
```

### 2.2 `object_type` 四类判别

`object_type` 表示真实业务世界中可独立识别、管理、判断、追踪或行动的对象。至少区分四类：

| 类型 | 定义 | 示例 | 建模要点 |
|---|---|---|---|
| 实体对象 | 真实事件中表现出来的人、事、物、组织、地点、产品、设备等稳定主体 | 客户、员工、供应商、产品、物料、仓库、设备 | 必须有稳定身份、核心属性、责任归属和可复用关系。 |
| 事件 / 记录对象 | 用来记录实体对象在某时点、场景或流程中形成的业务事实 | 库存记录、交易记录、质检记录、风险事件、工单、履约评估 | 重点记录发生时间、来源、状态、影响对象、证据和处理过程。 |
| 衍生 / 组成对象 | 某个主对象的组成部分、附件、明细、版本或派生事实 | 合同附件、订单明细、BOM 版本、库存批次、审批意见 | 只有可单独识别、维护、审批、追踪、复盘或形成关系时才升格为对象。 |
| 决策对象 | 决策本身需要被发起、确认、执行、跟踪和复盘的事实载体 | PO 发起决策、履约建议、替代料使用决策、风险处置决策 | 必须记录发起原因、输入依据、确认人、执行结果、状态变化和后续影响。 |

### 2.3 对象判定门禁

一个候选项至少满足两项，才适合成为对象：

- 有稳定身份，可被业务反复引用；
- 有生命周期或状态变化；
- 有责任人、归属方、管理部门或审批人；
- 参与多个业务关系；
- 会被确认、调整、追踪、写回、审计或复盘；
- 离开它，关键业务判断无法解释清楚。

默认不对象化：

- 编码：如 SKU，优先是 `产品.sku编码`；
- 数值 / 指标：如库存余额、可用数量、缺口数量；
- 状态 / 类型：如库存状态、需求类型、风险等级；
- 统计视图：如分仓分状态库存余额；
- 页面组件、报表行、查询结果；
- 算法输出：除非会被确认、追踪、失效、写回或复盘；
- 业务规则 / 口径：优先作为关系约束、指标 / 算子输入或治理策略。

### 2.4 典型边界示例

| 候选项 | 推荐处理 |
|---|---|
| 产品、物料 | 实体对象。`SKU` 是识别属性，不是对象名的一部分。 |
| 库存 | 多数场景是事件 / 记录对象，记录产品或物料在仓库、批次、状态、数量、时间上的存量事实。 |
| 库存数量 / 可用数量 | 属性、指标或计算结果，不是对象。 |
| 合同附件 | 默认是合同的衍生 / 组成对象；如果只是文件链接，可作为属性或文档引用。 |
| PO 发起 | 如果只是按钮，不是对象；如果要记录决策发起、审批、执行结果和后续跟踪，应建决策对象或行动执行记录。 |
| 替代规则 | 默认是 `物料 替代 物料` 的关系约束或替代覆盖逻辑；只有有版本、审批、生效期和发布废止时才对象化。 |

### 2.5 关系建模

关系不是 SQL join，必须表达两个对象之间稳定、可复用、有业务含义的联系。

每条关系必须说明：

- 连接哪两个对象；
- 业务含义是什么；
- 为什么这个判断、逻辑或行动需要它；
- 方向、数量约束、时间有效性或范围；
- 关系是否有属性，例如生效期、占比、置信度、来源系统、状态。

推荐表达：

```text
客户 需求 产品
产品 消耗 物料
库存 存放于 仓库
供应商 供应 物料
风险事件 影响 供应商
履约建议 针对 客户需求
```

## 三、动力层方法

动力层回答：

```text
业务世界如何计算、判断、变化和行动？
```

### 3.1 事实、指标、算子、行动的边界

动力层优先使用业务语言表达：事实、指标、算子、行动。BKN 元素名只用于说明其落位。

| 业务类型 | BKN 落位 | 判断标准 |
|---|---|---|
| 事实 | 对象自身的基础事实或状态 | 编号、名称、状态、数量、金额、负责人、发生时间等直接描述对象的内容；不应因为重要就升格为对象。 |
| 指标 | 关联到业务对象的稳定业务口径 | 有统一口径、周期、业务含义和责任边界；用于读取、比较、判断，不直接改变对象。 |
| 算子 | 关联到业务对象或对象集合的逻辑能力 | 用于计算、推理、排序、推荐、模拟、诊断或解释；输入事实、关系、规则、参数和上下文会影响输出，不直接改变对象。 |
| 行动 | 受控业务事务 | 会改变对象、影响对象、写回外部系统、触发审批、通知、单据或业务流程。 |

判断口诀：

```text
描述对象 → 事实；
读指标 → 指标口径；
算判断 → 逻辑能力；
改对象 / 影响外部系统 → 行动事务。
```

### 3.2 指标与算子表达

指标和算子都围绕业务对象工作，并且不改变对象本身。

指标通常表达对象上的稳定业务口径，例如交付率、风险分、可用量、履约率。每个指标必须说明：

- 关联到哪个业务对象；
- 业务含义、统计周期、口径边界和责任边界；
- 依赖哪些事实、关系或规则；
- 输出数值、状态或等级；
- 数据不足、口径冲突或权限不足时如何处理。

算子通常表达可调用的业务判断能力，例如影响分析、风险诊断、替代推荐、根因判断、履约模拟。每个算子必须说明：

- 服务哪个业务对象或对象集合；
- 完成什么判断、计算、推荐、模拟、诊断或解释；
- 输入事实、对象关系、业务规则、必要参数和上下文；
- 输出结构；
- 是否可被 Skill 调用、被行动前置校验引用，或被人工确认；
- 边界：数据不足、权限不足、模型不确定或规则冲突时如何处理；
- Eval 和 Trace 要求。

### 3.3 行动表达

行动会改变对象、影响对象或外部系统，本质上是受控业务事务，因此必须受前置条件、权限、风险和审计约束。

每个行动必须说明：

- 谁发起：用户、Agent、系统事件或定时任务；
- 改变什么：对象、事实、关系、状态或外部流程；
- 是否形成行动记录、决策记录或审批记录；
- 前置条件：数据、权限、状态、审批、窗口期；
- 参数绑定：来自对象、关系、用户输入还是常量；
- 影响范围：哪些对象、流程、用户或外部系统受影响；
- 风险类型和审批策略；
- 执行结果：成功、部分成功、失败或已回滚如何表达；
- 重复提交、并发冲突、外部系统失败如何处理；
- 是否支持 dry run、补偿、回滚和人工接管；
- 审计与 Trace 记录；
- 行动生命周期：`Proposed → Validated → Approved → Executed → Audited → Closed / Reverted`。

### 3.4 行动与工具调用的边界

行动是业务语义层定义，Tool、API、MCP 是技术执行层能力。

核心原则：

```text
行动说明业务上允许做什么；
Tool / API / MCP 说明技术上如何执行；
AI 不应绕过行动定义直接调用高影响工具。
```

行动应描述：

- 业务上要做什么；
- 对哪个对象做；
- 在什么条件下允许做；
- 需要哪些参数；
- 会产生什么影响；
- 是否需要审批、审计或回滚。

工具绑定只回答技术执行方式，不替代业务行动定义。

每次高影响工具调用必须归属于某个受控行动、只读查询或逻辑能力，并继承对应权限、审计、风险和 Trace 要求。不得把工具接口本身当成业务行动。

### 3.5 Prompt 外置规则

以下内容不应长期停留在 Prompt 中：

- 指标口径、统计周期、过滤条件；
- 风险评分、排序、推荐、模拟、影响路径；
- 行动前置条件；
- 工具参数绑定；
- 数据不足、权限不足、执行失败时的降级逻辑；
- 行动结果的状态记录、审计和回滚要求。

稳定、可复用、可测试的内容应沉淀为 BKN 指标、受控算子、行动定义或治理策略。

## 四、风险治理层方法

风险治理层回答：

```text
Agent 在什么事实、角色、目的、风险和审批边界内可以做？
```

### 4.1 风险识别

以下信号必须进入治理建模：

- 涉及客户、员工、合同、价格、资金、生产、风控等敏感对象；
- 行动会写回系统、发送外部通知、生成正式单据或改变主数据；
- Agent 使用 LLM 判断、摘要、分类或生成正式内容；
- 存在批量行动、周期调度或跨对象影响；
- 数据不足、质量不可信或来源冲突；
- 输出会影响承诺、采购、合同、资金、生产或客户沟通。

### 4.2 治理要素

每个高风险场景必须明确：

- `risk_type`：风险类型和业务后果；
- `control_scope`：策略管哪些对象、属性、逻辑、行动或 Skill；
- `control_policy`：读取、分析、建议、执行、审批、拒绝、脱敏、审计、回滚；
- 动态权限：谁以什么目的、在什么对象范围和关系上下文中、面对什么风险等级，可以读取、分析、建议、执行、审批或必须拒绝；
- Trace 要求：输入事实、关系、指标、算子、模型调用、工具调用、权限决策、审批、失败和人工修改。

动态权限应能形成可判定的控制结论：

```text
主体角色 + 业务目的 + 对象范围 + 关系上下文 + 行动类型 + 风险等级
→ 允许 / 脱敏 / 需审批 / 只读 / 拒绝
```

### 4.3 默认拒绝自动化边界

以下情况不得默认自动执行：

- 高风险写回、外部通知、资金、合同、采购、生产变更；
- 跨客户、跨组织、跨权限范围的数据或资源调配；
- 数据不足但业务后果重大；
- LLM 仅凭文本推断生成正式结论；
- 审批未通过或责任人未确认；
- 无法记录证据链或失败无法回滚。

## 五、Skill / Agent 应用层方法

Skill 是把 BKN 知识转化为 Agent 可执行能力的工程单元。

### 5.1 Skill 任务契约

一个 Skill 必须说明：

| 维度 | 要求 |
|---|---|
| 帮谁 | 业务角色、责任边界、权限范围。 |
| 做什么 | 判断、解释、建议、草案、申请、执行或追踪。 |
| 依赖什么 | 对象、关系、指标、算子、规则、输入材料和外部工具。 |
| 输出什么 | 结论、依据、原因、建议动作、待确认问题、报告或申请。 |
| 不能做什么 | 越权访问、无证据结论、高风险直执行、自动承诺。 |

不合格：

```text
做一个库存查询 Skill。
```

合格：

```text
做一个需求履约预检 Skill，帮助 PMC 在承诺交付前基于客户需求、产品、库存、BOM、替代关系和在途事实，判断是否可满足，输出缺口原因、建议动作和需要人工确认的问题；不得自动发起采购或改写库存。
```

### 5.2 Skill Card 必填项

Skill Card 是应用层消费契约，不是本体层本身。它用于说明 Skill 如何引用对象、关系、指标、算子、行动和治理策略，不能反向驱动本体边界。

每个 Skill 至少包含：

- `name`
- `business_goal`
- `user_persona`
- `trigger`
- `bkn_scope`
- `required_objects`
- `required_relations`
- `required_logic_properties`
- `callable_actions`
- `risk_policy`
- `required_permissions`
- `output_contract`
- `evaluation_criteria`
- `trace_required`

### 5.3 Skill 执行骨架

Skill 开发应按以下骨架组织：

```text
1. 识别用户意图、角色和任务边界
2. 校验权限、目的和输入完整性
3. 检索相关对象和关系
4. 读取指标或调用算子
5. 生成判断、原因、证据和建议
6. 如需行动，先生成 proposal 并校验 pre-conditions
7. 按 risk_policy 决定执行、审批、dry run 或拒绝
8. 输出符合契约的结果
9. 写入 Trace / Eval 所需证据
```

## 六、验证方法

验证必须覆盖正向、边界、拒绝、权限、数据不足和工具失败。

| 层级 | 验证重点 |
|---|---|
| 概念模型 | 对象是否真实、可识别、可追踪；误建对象是否被降级；关系是否有业务含义。 |
| 逻辑能力 | 指标口径是否稳定；算子输入、边界和输出契约是否正确。 |
| 行动 | 是否是受控业务事务；前置条件、审批、影响范围、执行结果、失败处理和回滚是否生效。 |
| 治理 | 无权限、敏感数据、高风险行动和数据不足时是否拒绝或降级。 |
| Skill / Agent | 是否完成角色任务，是否给出依据、原因、建议动作和待确认问题。 |
| Trace / Eval | 是否能还原事实来源、关系、指标、算子、权限决策、工具调用、审批、行动结果和输出依据。 |

需求发现阶段引用本表时，只需要提出可验收、可追溯来源、可留痕或需治理的业务要求；不得把 PRD 写成平台 Trace / Eval 设计文档。

验收用例至少包含：

- 典型用例；
- 边界用例；
- 权限 / 拒绝用例；
- 数据不足用例；
- 工具失败用例；
- 证据解释用例；
- 业务用户反证用例：证明哪些候选不应成为对象或行动。

## 七、建模质量检查原则

BKN 模型质量不只看结构完整，还要看是否能支撑 AI 工作。

### 7.1 对象质量

检查：

- 是否是业务对象，而不是技术表；
- 是否有稳定身份；
- 是否有清晰业务含义；
- 是否能被业务人员理解；
- 是否能承载属性、关系、逻辑或行动。

### 7.2 关系质量

检查：

- 是否表达业务语义；
- 是否方向清晰；
- 是否不是简单数据库 Join；
- 是否能支持 Skill 推理；
- 是否需要关系属性或关系对象化。

### 7.3 事实与逻辑质量

检查：

- 是否区分基础事实、业务状态、指标口径和推理结果；
- 编号、名称、类型、状态、日期、金额、负责人等是否只是对象事实，而不是对象本身；
- 指标、评分、推荐、诊断、预测和影响分析是否被识别为逻辑能力，而不是对象事实；
- 指标是否有统一口径；
- 算子是否是可调用能力，而不是临时 Prompt 推理。

### 7.4 行动质量

检查：

- 是否是真正业务行动；
- 是否是会改变对象、影响对象或外部系统的受控业务事务；
- 是否有作用对象；
- 是否有触发条件、前置条件和参数绑定；
- 是否有影响范围；
- 是否说明成功、部分成功、失败、回滚或人工接管；
- 是否有风险治理。

### 7.5 治理质量

检查：

- 高风险行动是否有审批；
- 是否有权限边界；
- 是否有审计要求；
- 是否需要 dry run；
- 是否需要回滚；
- 是否记录 Trace；
- 是否限制执行范围。

### 7.6 Skill 可用性

检查：

- Skill 是否基于 BKN 对象和关系工作；
- Skill 是否调用 BKN 中定义的指标或算子；
- Skill 是否通过受控行动定义建议或执行行动；
- Skill 是否受 `risk_type` 和 `control_policy` 约束；
- Skill 是否有 Eval；
- Skill 是否有 Trace。

## 八、核心反模式

应避免以下建模方式：

1. 把数据库表直接建成 `object_type`。
2. 把接口字段直接当成业务属性。
3. 把报表或 Dashboard 当成业务对象。
4. 用 `related_to` 表达所有关系。
5. 把复杂计算结果当成基础事实属性。
6. 把指标口径写死在 Prompt 中。
7. 把工具调用直接当成业务行动。
8. 让 AI 绕过受控行动定义调用高风险工具。
9. 高风险行动缺少审批和审计。
10. Skill 不声明可访问对象、可调用逻辑和行动边界。
11. Skill 没有 Eval。
12. Skill 没有 Trace。

## 九、总结

BKN 本体建模方法论可以概括为：

> 以业务对象和业务关系为基础，  
> 以指标和算子承载计算、推理和 AI 能力，
> 以受控行动表达会改变对象或影响外部系统的业务行动，  
> 以风险类型和控制策略约束 AI 行动边界，  
> 最终让 Skill 能够基于 BKN 理解业务、分析问题、建议行动、执行任务，并被评估和追踪。
